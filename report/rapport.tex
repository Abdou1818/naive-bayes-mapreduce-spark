\section{Objectif}

Implémenter le classifieur \textbf{Naive Bayes multinomial} selon le paradigme
\textbf{MapReduce} sur Apache Spark, en \textbf{deux versions} :
\begin{itemize}
  \item une version \textbf{RDD} (API bas niveau, \code{map} / \code{reduceByKey}) ;
  \item une version \textbf{DataFrames} (API relationnelle, \code{explode} + \code{groupBy} / \code{agg}).
\end{itemize}
Les deux versions doivent produire des \textbf{prédictions rigoureusement identiques}, et
l'on mène une \textbf{étude de scalabilité} faisant varier (i) le volume de données et
(ii) le parallélisme (\code{local[k]}).

\section{Rappel du modèle}

On travaille en \textbf{sac de mots} (bag-of-words). Pour un document $d$, une classe $c$,
un vocabulaire de taille $V$ et un lissage de Laplace $\alpha = 1$, le modèle multinomial
pose :
\begin{align}
  \log P(c)      &= \log\!\left( \frac{N_c}{N} \right), \\
  \log P(w \mid c) &= \log\!\left( \frac{\mathrm{count}_{w,c} + \alpha}{\mathrm{total}_c + \alpha V} \right),
\end{align}
où $N$ est le nombre de documents d'entraînement, $N_c$ les documents de la classe $c$,
$\mathrm{count}_{w,c}$ les occurrences du mot $w$ dans la classe $c$, et
$\mathrm{total}_c = \sum_w \mathrm{count}_{w,c}$.

La \textbf{prédiction} maximise, en logarithme (pour éviter les sous-débordements) :
\begin{equation}
  \mathrm{score}(d, c) = \log P(c) + \sum_{w \in d} \mathrm{count}_w(d)\, \log P(w \mid c).
\end{equation}

Notre implémentation reproduit exactement \code{sklearn.naive_bayes.MultinomialNB}
($\alpha=1$, \code{fit\_prior=True}), ce qui est \textbf{vérifié empiriquement} (§\ref{sec:valid}).

\section{Formulation MapReduce}

L'entraînement se réduit à des \textbf{comptages} --- le cas idéal pour MapReduce. Le cœur
est le calcul de $\mathrm{count}_{w,c}$ (occurrences de chaque mot par classe).

\subsection{Version RDD (\texttt{src/nb\_rdd.py})}
Chaque document est une paire \code{(indice_classe, [indices de mots])}.
\begin{itemize}
  \item \textbf{MAP} (\code{flatMap}) : pour chaque occurrence d'un mot $w$ dans un document
        de classe $c$, émettre \code{((c, w), 1)}.
  \item \textbf{REDUCE} (\code{reduceByKey(add)}) : additionner les $1 \to \mathrm{count}_{w,c}$.
\end{itemize}
On compte de même les documents par classe ($N_c$) et les tokens par classe
($\mathrm{total}_c$).

\subsection{Version DataFrames (\texttt{src/nb\_dataframe.py})}
Le DataFrame a le schéma \code{(label:int, words:array<int>)}.
\begin{itemize}
  \item \textbf{MAP} : \code{explode(words)} $\to$ une ligne \code{(label, word)} par
        occurrence (équivalent du \code{flatMap}).
  \item \textbf{REDUCE} : \code{groupBy(label, word).count()} $\to \mathrm{count}_{w,c}$.
\end{itemize}

\subsection{Construction du modèle et broadcast}
Les deux versions produisent \textbf{exactement les mêmes agrégats entiers}, puis appellent
la \textbf{même} fonction \code{nb_common.build_model} (comptes $\to$ log-probabilités
lissées) et la \textbf{même} fonction \code{nb_common.predict_indices}. Comme les comptes
sont entiers (reproductibles au bit près) et que la prédiction est déterministe, \textbf{les
deux versions donnent des prédictions identiques par construction}.

Le modèle est stocké de façon \textbf{creuse mais équivalente au dense} : on ne mémorise
$\log P(w\mid c)$ que pour les couples $(c, w)$ de compte non nul, plus une valeur par défaut
$\log(\alpha / (\mathrm{total}_c + \alpha V))$ pour les mots de compte nul. Ce modèle est
\textbf{broadcasté} (\code{sc.broadcast}) : Spark l'envoie une seule fois par worker (lecture
seule), au lieu de le re-sérialiser à chaque tâche --- essentiel quand $V \times C$ est grand.
La prédiction se fait via un \code{map} (RDD) ou une \textbf{UDF} (DataFrame) référençant le
modèle broadcasté.

\section{Protocole expérimental}

\begin{itemize}
  \item \textbf{Machine} : macOS (Apple Silicon, arm64), \textbf{8 cœurs} (4 \emph{performance}
        + 4 \emph{efficiency}).
  \item \textbf{Pile} : Python 3.12, PySpark 3.5.8, JDK 17. Spark en mode \textbf{local}
        (\code{local[k]}).
  \item \textbf{Deux expériences} (\code{src/benchmark.py}) :
  \begin{enumerate}
    \item \textbf{Volume} : à \code{local[*]} (8 threads), réplication de 20 Newsgroups par un
          facteur $\in \{\times 1, \times 2, \times 4, \times 8\}$.
    \item \textbf{Parallélisme} : à volume fixe (facteur $\times 8$, $\sim$121\,k documents),
          variation de \code{local[k]}, $k \in \{1, 2, 4, 8\}$.
  \end{enumerate}
  \item \textbf{Précautions méthodologiques} (corrigées après une première version) :
  \begin{itemize}
    \item \textbf{Anti-fuite de données} : la réplication est faite \textbf{après} le
          découpage train/test, et \textbf{séparément} de chaque côté. Un document de test ne
          peut donc jamais apparaître (même dupliqué) dans l'entraînement.
    \item \textbf{Bruit de mesure} : option \code{--repeat 3} --- chaque point est mesuré 3
          fois et agrégé en \textbf{moyenne $\pm$ écart-type} (colonnes \code{train_time_std},
          \code{predict_time_std} ; barres d'erreur sur les graphes).
  \end{itemize}
\end{itemize}

\begin{lstlisting}
python src/benchmark.py --factors 1 2 4 8 --cores 1 2 4 8 --repeat 3
\end{lstlisting}

        parallélise déjà en interne ; l'overhead SQL + UDF domine).
\end{itemize}

\section{Discussion}

\subsection{RDD vs DataFrames}
Sur ce problème (comptages + prédiction par UDF), \textbf{RDD est plus rapide} que
DataFrames, surtout en prédiction. L'API DataFrames reste plus lisible et bénéficierait
davantage à plus grande échelle ou avec des fonctions natives Spark (sans UDF Python). Le
point clé pédagogique --- \textbf{prédictions identiques} --- est vérifié.

\subsection{Speed-up sous-linéaire}
Le mode \code{local} fait tourner driver et exécuteurs dans \textbf{une seule JVM} ;
« cœurs » $=$ threads. Les overheads (shuffle, sérialisation, warmup, GC) et l'hétérogénéité
des cœurs Apple Silicon expliquent un speed-up \textbf{bien en deçà du linéaire idéal}
($\sim 2\times$ utiles sur 4 cœurs performance).

\subsection{Accuracy et réplication --- un effet de lissage}
Après correction de la fuite, l'accuracy \textbf{ne monte plus fortement} avec le volume,
mais elle \textbf{n'est pas parfaitement constante} ($0.620 \to 0.711$). Ce n'est
\textbf{pas} de la fuite : répliquer le \emph{train} multiplie tous les comptes par le
facteur $f$, si bien que $\log P(w\mid c) = \log\big((f\cdot\mathrm{count}+\alpha)/(f\cdot
\mathrm{total}+\alpha V)\big)$ --- le lissage $\alpha=1$ devient \textbf{relativement
négligeable} quand les comptes gonflent, donc le modèle est moins régularisé. La valeur de
référence « honnête » est celle à $\times 1 = 0.620$. Pour une accuracy strictement plate, il
faudrait mettre à l'échelle le lissage ($\alpha = f$) --- mais l'étude visant les
\textbf{temps}, on conserve $\alpha = 1$ et l'on documente l'effet.

% ======================================================================
\section{Limites}

\begin{itemize}
  \item \textbf{Mode local uniquement} : pas de vrai cluster ; conclusions valables pour un
        nœud multi-cœurs.
  \item \textbf{Cœurs hétérogènes} (Apple Silicon) : le point \code{local[8]} est peu fiable.
  \item \textbf{Prédiction DataFrame par UDF Python} : goulot d'étranglement.
  \item \textbf{Réplication} comme proxy de « big data » : gonfle le volume mais pas la
        diversité ; interagit avec le lissage.
\end{itemize}

% ======================================================================
\section{Conclusion et pistes}

Le classifieur Naive Bayes multinomial a été implémenté en MapReduce sur Spark en deux
versions \textbf{cohérentes} (prédictions identiques, alignées sur scikit-learn), avec une
étude de scalabilité rigoureuse (anti-fuite, mesures répétées). Les résultats montrent un
\textbf{scaling en volume sous-linéaire}, un \textbf{avantage de la version RDD}, et un
\textbf{speed-up parallèle réel mais modéré} limité par le mode local et le matériel.

Pistes : exécution sur cluster (YARN/K8s), prédiction DataFrame sans UDF Python, métriques
par classe (précision/rappel, matrice de confusion), comparaison temps Spark vs scikit-learn
sur petit volume, mise à l'échelle du lissage.
